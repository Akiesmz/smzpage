Learning Assistant 背后的思考：为什么我要做一个学习助手关于我用 TypeScript 构建 Learning Assistant 项目的完整复盘：从需求分析到技术选型，从第一个 commit 到第一次部署。
Learning Assistant 这个项目的想法来源于一个简单的观察：学习这件事，最大的敌人不是「没有好资源」，而是「没有好节奏」。很多人不是不想学习，而是不知道今天应该学什么、学多少、学到什么程度。他们需要的不是一个更多的课程平台，而是一个能帮他们管理学习节奏的助手。为什么用 TypeScript选择 TypeScript 而不是 Python 或其他语言，是因为我希望这个项目的核心是一个前端体验——它应该是一个用户每天打开、每天使用的工具，而不是一个后台运行的脚本。TypeScript 给了我类型安全、良好的 IDE 支持和丰富的前端生态。另一个原因是我想把这个项目作为一个「前端工程实践」的载体。TypeScript + React 的组合让我可以实践组件化设计、状态管理、类型驱动开发等现代前端技术。Learning Assistant 不只是一个产品，也是我提升前端技能的练习场。核心设计思路Learning Assistant 的核心不是「教用户学什么」，而是「帮用户管理学习状态」。它追踪你最近学了什么、哪些知识点需要复习、今天的精力适合学新内容还是复习旧内容。这种「状态感知」是它区别于其他学习工具的关键。这个设计思路来自于「间隔重复」的研究。认知科学告诉我们，人脑的记忆曲线是指数衰减的——学到一个新知识后，如果不复习，遗忘速度会越来越快。但如果在合适的时间点复习，就可以用最少的时间维持最强的记忆。Learning Assistant 就是在帮用户找到那些「合适的时间点」。技术实现中的取舍在实现过程中，我做了几个关键的技术取舍。第一个是数据存储：我选择了本地存储而不是云数据库。这降低了功能复杂度，但也意味着用户的数据只存在于本地设备上。对于一个学习工具来说，这是一个合理的权衡——隐私优先，但牺牲了跨设备同步。第二个是 AI 集成：我在项目中加入了 AI 辅助功能，用来分析学习模式和推荐复习计划。但这不是核心功能——它是一个「增强」，而不是「依赖」。即使没有 AI，用户也能手动管理自己的学习节奏。这种「AI 可选」的设计让产品更健壮，也更容易维护。学到的教训这个项目教会我最重要的一件事是：产品的价值不在于功能多少，而在于用户是否愿意每天使用它。Learning Assistant 的功能很少——远少于市面上的大多数学习工具。但它的每一个功能都是围绕「用户每天打开它」这个核心目标设计的。另一个教训是：不要过早优化。我花了太多时间在第一版的 UI 细节上，而忽略了核心功能的可用性。后来我把 UI 的优先级降低了，把精力集中在学习追踪和复习提醒上，产品的体验反而好了很多。
